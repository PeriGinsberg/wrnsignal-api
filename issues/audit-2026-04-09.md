# Scoring Engine Audit — 2026-04-09

Systematic review of extraction, family classification, gates/requirement units,
and bullet generation / scoring logic. Generated from four parallel subagent
reviews of the scoring codebase after a day of reactive bug-hunting exposed
systematic weak spots across multiple candidates.

**Scope**: `app/api/jobfit/extract.ts`, `app/api/jobfit/scoring.ts`,
`app/api/jobfit/constraints.ts`, `app/api/jobfit/bulletGeneratorV5.ts`,
`app/api/jobfit/deterministicBulletRendererV4.ts`,
`app/api/profile-intake/route.ts`, `app/api/_lib/jobfitProfileAdapter.ts`,
`app/api/jobfit/route.ts`.

**How to use this doc**: Each finding has a severity. Work through **P0** first —
those will definitely produce wrong output on real candidates. Then **P1**
(likely in edge cases). **P2** items are theoretical concerns worth noting but
not urgent.

**Definitions**:
- **P0** — Will definitely fire on common candidates and produce wrong output
- **P1** — Likely to fire on edge cases we haven't tested yet
- **P2** — Theoretical concern, may not fire but worth knowing

## P0 — Fix first (8 items)

### P0-1 — `inferTargetFamilies` drift: profile-intake pushes "Design" but no such JobFamily exists
**File**: `app/api/profile-intake/route.ts:345`
**Symptom**: profile-intake's `inferTargetFamilies` pushes `"Design"` for creative/design target roles, but `JobFamily` in `signals.ts:10` has no `"Design"` member. The string round-trips through Supabase but downstream code that type-checks against JobFamily will either TypeScript-error at compile time or produce garbage classification at runtime.
**Fix**: Either add `"Design"` to the JobFamily type and wire it through the scoring tags, OR remove the push. The subagent flagged this as a P0 because it's a latent type bomb.

### P0-2 — profile-intake missing Legal, Healthcare, Trades family mappings
**File**: `app/api/profile-intake/route.ts:319-364`
**Symptom**: Candidates targeting "lawyer", "nurse", "electrician", "plumber" never get any family assignment from profile-intake and fall through to `"Other"`. The adapter version (`_lib/jobfitProfileAdapter.ts:120`) DOES handle some of these (Engineering covers medical device), but the legal/healthcare/trades paths exist in `JobFamily` and aren't populated.
**Fix**: Add checks for legal keywords (`lawyer`, `attorney`, `legal`, `law`), healthcare keywords (`nurse`, `physician`, `clinical`, `healthcare`), and trades keywords (`electrician`, `plumber`, `welder`, `hvac`). See consolidation fix below.

### P0-3 — Job family cascade: `engineering_technical` and `software_it` both weighted +8
**File**: `app/api/jobfit/extract.ts:1472-1473`
**Symptom**: Both tags contribute +8 to their respective families. When a JD has both (common for Software Engineer roles), Engineering and IT_Software tie at 16, and cascade order picks Engineering first. Software engineers routing to Engineering is wrong — they should route to IT_Software.
**Fix**: Either boost `software_it` to +9, OR reorder the title cascade so `jobTitleIsSoftware` is checked before `jobTitleIsEngineering`, OR add a tiebreaker that prefers IT_Software when the title contains "software" / "developer".

### P0-4 — `finance_corp` and `accounting_finops` tied at +5, Finance wins by accident
**File**: `app/api/jobfit/extract.ts:1445-1446`
**Symptom**: Both tags contribute +5. Cascade order puts Finance after Accounting, so ties go to Finance. A Senior Accountant JD mentioning "financial analysis" will misclassify as Finance.
**Fix**: Change `accounting_finops` to +6 (stronger signal — "accounting operations" is domain-specific), or lower `finance_corp` to +4.

### P0-5 — V4 fallback template says "internship or early role" on full-time jobs
**File**: `app/api/jobfit/deterministicBulletRendererV4.ts:493-496`
**Symptom**: The `executionBridge()` function hardcodes "For an internship or early role, this is enough to show you can step into [capability] and get up to speed quickly." This string appears in candidate-facing bullets for full-time roles when V5 falls back to V4 (which we now know can happen silently even with healthy API credits, if max_tokens truncates mid-JSON).
**Fix**: Pass the job signal (`isInternship`, `isSummer`) to the bridge function and branch the text. If not internship, use "You have shown [X], which translates to [capability] in a production environment."

### P0-6 — V4 fallback template literal "the work this role requires"
**File**: `app/api/jobfit/deterministicBulletRendererV4.ts:272, 508, 523`
**Symptom**: When `capabilityPhrase()` and `summarizeJobFact()` can't match any of ~50 hardcoded patterns, they return the literal string `"the work this role requires"`. This appears in candidate bullets as "this role relies on the work this role requires, and you already have experience doing that" — which was the exact garbage the user saw on Reece's Pharma Sales Rep test.
**Fix**: Return `null` when no concrete phrase matches, so the bullet gets dropped entirely rather than rendered with placeholder garbage. Let V5 handle the case properly.

### P0-7 — Garbage filter regex `/^position\b/` flags legitimate titles starting with "Position"
**File**: `app/api/jobfit/route.ts:495-498` (fresh path), also in the cache-hit path around line 339
**Symptom**: Already fixed today, but the subagent confirmed that the filter needs to stay `/^word$/` and not regress to `/^word\b/`. Re-document in tests.
**Status**: FIXED today in commit `f7fd2327`. Flagging for regression test coverage.

### P0-8 — Duplicate risk emission: same evidence fires WHY + RISK simultaneously
**File**: `app/api/jobfit/scoring.ts:1227-1272`
**Symptom**: A "near miss" match (coverage below `minimumCoverage` but above `nearMissFloor`) is selected as a `WHY_EXECUTION_PROOF` at weight ~85 AND as a `RISK_MISSING_PROOF` at weight 0 simultaneously, using the same profile_fact. Ryan's Pharma Sales Rep test showed this: `territory_execution` fired as a positive WHY with "Managed 400 B2B prospects" and as a negative RISK with the same profile_fact on the same evidence.
**Fix**: In `selectWhyMatches()`, exclude matches where the same job unit is already flagged in a gap risk. Or, when a near-miss fires as a gap, remove it from the WHY candidate pool.

### Family classification drift (MANDATORY to consolidate)

### P1-1 — Three copies of `inferTargetFamilies` have materially different logic
**Files**:
- `app/api/profile-intake/route.ts:319-364`
- `app/api/_lib/jobfitProfileAdapter.ts:120-304`
- `app/api/jobfit/extract.ts:2346-2349` (different: tag-based wrapper)

6. extract.ts's `inferTargetFamiliesFromTags` returns ONLY ONE family (wrapped in array), while profile-intake and adapter return up to 4

**Fix**: Consolidate into a single shared module (`lib/jobfit-family-inference.ts`) imported everywhere. See consolidation section at bottom of this doc.

### P1-2 — `amKeywords` asset_management regex fires on company boilerplate
**File**: `app/api/jobfit/extract.ts:2427` (known pattern: UBS test)
**Symptom**: `"asset management"` in firm About Us text ("we offer investment banking, asset management, and wealth management services") fires `financeSubFamily = "asset_management"` for wealth management Client Associate roles. Same bug class as the IB boilerplate leakage we fixed today.
**Fix**: Apply the same `ibRoleContext` pattern — require "asset management" to be adjacent to a role noun (analyst, associate, PM, team, group, division) OR combine with `hasPortfolioAnalysis`/`hasProspecting` requirement unit check.

### P1-3 — Profile `ibKeywords` uses bare "deal" and "coverage"
**File**: `app/api/jobfit/extract.ts:2458` (`inferProfileFinanceSubFamily`)
**Symptom**: Candidates with "led deal strategy" (corporate development) or "coverage models" (consulting) get misclassified as IB because the profile-side IB detection uses bare `"deal"` and `"coverage"` keywords.
**Fix**: Apply the same role-context requirement we added to job-side detection. Bare "deal" and "coverage" should only fire with IB-specific context.

### P1-4 — adapter's Consulting check has bare `"strategy"` that over-matches
**File**: `app/api/_lib/jobfitProfileAdapter.ts:160`
**Symptom**: "Product Strategy Manager", "Growth Strategy", "Marketing Strategy" all get pushed to Consulting because of the bare `"strategy"` substring check. Product strategy is Marketing, not Consulting.
**Fix**: Require "strategy" to be compound (`"strategy and operations"`, `"strategy consulting"`, `"business strategy"`) or co-occur with another Consulting signal.

### P1-5 — `inferYearsExperienceApprox` off-by-one: span calculation is exclusive not inclusive
**File**: `app/api/jobfit/extract.ts:~2295`, mirrored in `app/api/profile-intake/route.ts:~200`
**Symptom**: `totalMonths += curEnd - curStart` computes the span between month indices, not the inclusive count. Jan 2024 – Dec 2024 (should be 12 months) is computed as 11 months. Rounds down to 0 years in some cases.
**Fix**: Add +1: `totalMonths += (curEnd - curStart) + 1`. Verify rounding behavior doesn't change for existing test cases.

### P1-6 — Role-signal fallback returns 0 on a single role keyword
**File**: `app/api/jobfit/extract.ts:2313-2315`, `app/api/profile-intake/route.ts:~250`
**Symptom**: `if (roleSignals >= 1) return 0` bucket means a candidate with a single detected role keyword (e.g., their resume mentions "manager" once) returns 0 years experience, even if they have decades.
**Fix**: Change threshold to `if (roleSignals === 0) return null` so a single signal leaves the result unknown rather than zeroed.

### P1-7 — Profile grad year detection fires on "class of" text in cover letters
**File**: `app/api/jobfit/extract.ts:inferProfileGradYear`
**Symptom**: Not confirmed yet, but we should verify that `inferProfileGradYear` only reads the resume/profile text and never reads external text like cover letter drafts or writing samples that may contain references to other class years.
**Fix**: Scope the detection to the resume section only.

### P1-8 — CDL regex has unescaped dot: `commercial driver.s license`
**File**: `app/api/jobfit/extract.ts:2814`
**Symptom**: The regex `/commercial driver.s license/i` uses literal `.` which matches any character. "commercial driverX license" would match. Harmless in practice but sloppy.
**Fix**: Change to `commercial driver[''.]s license` to match both apostrophe forms.

### P1-9 — Support Associate title override missing trainee roles
**File**: `app/api/jobfit/extract.ts:2934`
**Symptom**: The credential-sponsorship title override handles "Client Associate", "Service Associate", etc., but MISSES trainee roles that are ALSO universally post-hire licensed:
- Insurance Agent trainee
- Mortgage Banker trainee (NMLS-regulated)
- Real Estate Agent trainee
- Registered Investment Advisor trainee

**Fix**: Extend the regex:
```ts
/\b(client associate|service associate|...|insurance agent trainee|trainee insurance|mortgage banker trainee|trainee.*mortgage|real estate agent trainee|trainee real estate|registered investment advisor trainee)\b/i
```

### P1-10 — CPA/Teaching/PE/CDL sponsor-detection fallbacks use bare single-word keywords
**File**: `app/api/jobfit/extract.ts:2944-2956`
**Symptom**: When POLICY's credential keywords are empty, the code falls back to single-word fallbacks like `["cpa"]`, `["teaching"]`, `["pe license"]`, `["cdl"]`. Single-word keywords are too broad — "teaching" matches "teaching opportunity", "pe" matches "PE class", etc.
**Fix**: Use compound fallback keywords only:
```ts
["cpa certification", "cpa license", "cpa required"]
["teaching certificate", "teaching credential"]
["pe license", "professional engineer license"]
["cdl required", "commercial driver"]
```

### P1-11 — Field-mismatch gate uses company-derived family, not role-derived family
**File**: `app/api/jobfit/constraints.ts:62-72`
**Symptom**: The gate blocks Business candidates from Technical jobs. But `job.jobFamily` might be auto-inferred from company industry, not from the actual role. A Marketing student applying for Product Marketing Manager at an engineering company should NOT be blocked just because the company is engineering.
**Fix**: Verify that `job.jobFamily` is derived from the role title/description extraction, not from company metadata. If company-derived, either separate them or require both to be technical before firing the gate.

### P1-12 — MBA gate has no exemption for MS Finance / MS Analytics candidates
**File**: `app/api/jobfit/constraints.ts:74-76`
**Symptom**: `if (job.mbaRequired) return force_pass`. No exemption for MS Finance, MS Analytics, or joint degree programs. Many MBA-preferred roles accept these as equivalents.
**Fix**: Add exemption: if profile has `function_tags` including `finance_corp` or `consulting_strategy` AND candidate is actively in a graduate program, soften to `floor_review` instead of hard force_pass.

### P1-13 — V5 silent fallback: max_tokens 4096 can still truncate pre-med candidates
**File**: `app/api/jobfit/bulletGeneratorV5.ts:200-204`
**Symptom**: We bumped from 2048 to 4096 today, but pre-med candidates with 28+ profile evidence units may still hit the cap. The catch block at line 243 catches all parse failures identically without logging token usage.
**Fix**: Bump to 6000 or 8000, AND log `json.usage.output_tokens` vs `max_tokens` after every call. If output_tokens reaches max, log a warning and surface it in debug.

### P1-14 — Risk emission duplicates: `RISK_ROLE_ARCHETYPE` pushed twice
**File**: `app/api/jobfit/scoring.ts:967-987`
**Symptom**: Same risk pushed as both a penalty (lines 967-978) AND as a risk-only code (lines 979-985). The `dedupeRiskCodes` at line 1333 keys on `${code}|${job_fact}|${profile_fact}|${risk}` — if the risk text differs between the two pushes, dedup fails and the user sees two identical-looking risks with different weights.
**Fix**: Pick one — either emit as penalty (auto-surfaces as risk via `capped.map(p => p.risk)`) OR as risk-only. Not both.

### P1-15 — Risk emission duplicates: `RISK_CONTENT_ROLE_CONFLICT` pushed twice
**File**: `app/api/jobfit/scoring.ts:994-1012`
**Symptom**: Same pattern as P1-14.
**Fix**: Same — pick one emission path.

### P1-16 — Implicit penalties via `shape.boost` don't emit explicit risks
**File**: `app/api/jobfit/scoring.ts:287-338`
**Symptom**: Many requirement units have explicit keyword checks that apply a negative `boost` (e.g., `-8`, `-16`) when the magic keywords are missing from profile evidence. This silently lowers the coverage score without emitting a visible `RISK_MISSING_PROOF`. The candidate sees a lower score with no explanation.
**Fix**: After evidence matching, scan for matches with `boost < -5` and emit a corresponding `RISK_MISSING_PROOF` flagging the implicit gap. The user sees both the score drop AND the reason.

### P1-17 — Decision thresholds in `decision.ts` are hardcoded and drift from POLICY
**File**: `app/api/jobfit/decision.ts:7-11`, `app/api/jobfit/policy.ts:125`
**Symptom**: `decision.ts` has hardcoded `96 / 75 / 60` but `policy.ts` has `{ apply: 77, review: 65 }`. These don't match. If you update POLICY, decision logic doesn't change.
**Fix**: Replace hardcoded values in decision.ts with `POLICY.thresholds.apply`, `POLICY.thresholds.review`, and add `POLICY.thresholds.priorityApply` for 96.

### P1-18 — Tool expansion for Google Workspace adds canonical tool names not in TOOL_ALIASES
**File**: `app/api/profile-intake/route.ts:313` and `app/api/jobfit/extract.ts`
**Symptom**: "Google Workspace" detection adds "google sheets", "google docs", "google slides" to the tools list, but these canonical names don't have entries in `TOOL_ALIASES`. A candidate who writes "Google Sheets" explicitly matches via alias lookup, while a candidate who writes "Google Workspace" gets the expansion. Asymmetric handling.

**Symptom**: Matches "cos " at word boundaries but could also match within larger strings unintentionally. The adapter's version correctly uses `\bcos\b`.

## P2 — Worth knowing (10 items)

### P2-1 — `inferYearsExperienceApprox` explicit match accepts up to 99 years, caps at 50
**File**: `app/api/jobfit/extract.ts:~2220`
**Symptom**: Regex allows `\d{1,2}` (up to 99) but the code clamps to 50. A resume saying "99+ years experience" silently becomes 50.
**Fix**: Restrict the regex to 1-50 years or log a warning.

### P2-2 — Location regex misses non-ASCII city names
**File**: `app/api/jobfit/extract.ts:~2536`
**Symptom**: `/^[A-Z][a-zA-Z\s.'-]+,\s*[A-Z]{2}\b/` doesn't match São Paulo, Zürich, Montréal, etc.
**Fix**: Extend character class to include Unicode letters.

### P2-3 — `includesPhrase` word-boundary check fails for special-character phrases
**File**: `app/api/jobfit/extract.ts:~1253`
**Symptom**: Phrases like `"fp&a"` embedded in larger strings don't match because `\W` boundary handling is inconsistent with `&`.
**Fix**: Use `\b` explicitly for alphanumeric-only phrases.

### P2-4 — `extractToolMentions` doesn't canonicalize tools found only via POLICY
**File**: `app/api/jobfit/extract.ts:~1388`
**Symptom**: Tools in POLICY but not in TOOL_ALIASES get added without canonical normalization.
**Fix**: Ensure all POLICY tools also have TOOL_ALIASES entries, or make `canonicalTool` handle the missing case.

### P2-5 — Bare-word detection in requirement units (scan all phrases)
**Files**: `app/api/jobfit/extract.ts` — requirement unit definitions
**Symptom**: The audit found many bare single-word phrases that are false-positive bombs:
- `analysis_reporting`: "analysis", "reporting", "dashboard", "metrics", "forecast"
- `performance_optimization`: "optimize", "conversion", "acquisition", "retention"
- `operations_execution`: "operations", "process", "schedule", "workflow"
- `prospecting_pipeline_management`: "pipeline" (matches engineering pipeline, data pipeline)
- `product_training_enablement`: "training", "demo"
- `med_device_industry_knowledge`: "industry knowledge" (too generic)

**Fix**: Systematic review of every profile_phrases / job_phrases array. Replace bare words with compound phrases that require domain context. See full list below in Appendix A.

### P2-6 — Jobtitle regex constants are 234 characters long, unmaintainable
**File**: `app/api/jobfit/extract.ts:~2625` (`jobTitleIsFinance`)
**Symptom**: Single-line regex with 30+ keywords, impossible to review diffs or add new keywords safely.
**Fix**: Refactor to arrays:
```ts
const FINANCE_TITLE_KEYWORDS = ["finance intern", "financial analyst", ...]
const jobTitleIsFinance = new RegExp(`\\b(${FINANCE_TITLE_KEYWORDS.join("|")})\\b`, "i").test(jobTitleSlice)
```

### P2-7 — Near-miss coverage handling is silently wrong
**File**: `app/api/jobfit/scoring.ts:463-468`
**Symptom**: Near-miss evidence (below `minimumCoverage`, above `nearMissFloor`) is filtered out of adequate-only checks but selected by `selectWhyMatches` without regard to adequacy. Root cause of P0-8 above.
**Fix**: Track coverage status in `WhyEvidenceMatch` and exclude from WHY selection when near-miss.

### P2-8 — monthMap null-coalescing defaults to January silently
**File**: `app/api/jobfit/extract.ts:~2250`
**Symptom**: `monthMap[m[1].toLowerCase()] ?? 0` returns January if regex capture doesn't match a valid month. Could mask bugs if regex and monthMap drift out of sync.
**Fix**: Validate and skip the range if month is unrecognized instead of defaulting.

### P2-9 — Seniority gate asymmetric: no overqualification flag
**File**: `app/api/jobfit/constraints.ts:175-186`
**Symptom**: Gate fires if job wants 5+ years and candidate has ≤2. No gate the other direction — a 10-year candidate applying to entry-level roles isn't flagged as overqualified.
**Fix**: Add symmetric check, emit as `floor_review` (not hard gate) for overqualification.

### P2-10 — WHY selection ranks by weight+kind-bonus but hard-codes code mapping first
**File**: `app/api/jobfit/scoring.ts:414-421, 509`
**Symptom**: Execution kind → `WHY_EXECUTION_PROOF` (code 4), tool kind → `WHY_TOOL_PROOF` (code 1). Codes are assigned before weight is considered. A low-weight execution match gets the same code as a high-weight one, which affects downstream selection tiebreakers.
**Fix**: Rank purely by weight without code-preference tiebreakers.

## Appendix A — Bare-word phrase risks in requirement units

This is a list of every single-word or generic phrase in profile_phrases / job_phrases
arrays that is a false-positive risk. Each should be replaced with a compound phrase
or have a companion context requirement.

Format: `[key] "phrase" — what it incorrectly matches → suggested compound form`

### `analysis_reporting`
- `"analysis"` — matches "competitive analysis", "job analysis", "variance analysis" → require prefix "data analysis" / "performance analysis" / "financial analysis"
- `"reporting"` — matches "status reporting", "budget reporting" → pair with "dashboard" / "metrics" / "data"
- `"dashboard"` — matches any "dashboard" in any function → pair with analytics context
- `"metrics"` — matches "success metrics" in any function → require "track metrics" / "analyze metrics"
- `"forecast"` — matches "weather forecast", "sales forecast" → require "financial forecast" / "forecast model"

### `performance_optimization`
- `"optimize"` — matches "optimize process", "optimize staffing" → require marketing channel context
- `"conversion"` — matches "lead conversion", "religious conversion" → require "conversion rate" / "conversion funnel"
- `"acquisition"` — matches "talent acquisition", "company acquisition" → require "customer acquisition" / "user acquisition"
- `"retention"` — matches "employee retention", "data retention" → require "customer retention" / "user retention"

### `operations_execution`
- `"operations"` — matches IT ops, hospital ops, every function → pair with "process" / "execution" / "management"
- `"process"` — matches approval process, design process → pair with "improvement" / "optimization"
- `"schedule"` — matches schedule appointments, schedule staff → pair with "staff" / "events"
- `"workflow"` — matches any workflow → pair with "execution" / "automation"

### `prospecting_pipeline_management`
- `"pipeline"` — matches data pipeline, oil pipeline, deployment pipeline → pair with "sales" / "deals" / "opportunities"
- `"outreach"` — matches community outreach, event outreach → require "sales outreach" / "business outreach"

### `product_training_enablement`
- `"training"` — matches compliance training, diversity training, any training → require "product training" / "staff training" / "customer training"
- `"demo"` — matches system demo, demo environment → require "product demo" / "product demonstration"

### `med_device_industry_knowledge`
- `"industry knowledge"` — matches any industry → require "medical device industry knowledge" / pair with "medical" / "device"

### `client_commercial_work`
- `"client"` — matches every business job → rely on `minMatches: 2` gate, OR require "client AND (relationship OR success OR communication)"

### `coaching_instruction_facilitation`
- `"coach"` / `"coaches"` — matches "coach someone on Excel" → pair with "sports" / "athletes" / "training"

### `drafting_documentation`

- `"wrote"` — matches any writing → pair with "report" / "memo" / "brief"
